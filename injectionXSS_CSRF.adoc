== Introduction théorique

Contrairement aux Injections SQL qui visent le serveur (vol de base de données), le **Cross-Site Scripting (XSS)** vise les **utilisateurs** du site.

Le principe est simple : l'attaquant injecte du code **JavaScript** malveillant dans une page web. Ce code sera ensuite exécuté par le navigateur de la victime comme s'il faisait légitimement partie du site.

=== Les 3 types de XSS
1.  **Reflected XSS (Réfléchi) :** L'attaque passe par un lien piégé. Le serveur "reflète" l'injection immédiatement (ex: un message d'erreur ou une recherche). C'est temporaire.
2.  **Stored XSS (Stocké) :** L'attaque est enregistrée sur le serveur (ex: commentaire, profil). Chaque visiteur de la page sera infecté. C'est la forme la plus dangereuse.
3.  **DOM-based XSS :** L'attaque se passe uniquement côté client (dans le navigateur), souvent à cause d'un mauvais script JS qui manipule l'URL sans passer par le serveur.

=== Cross-Site Scripting (XSS)
* **La faille :** Le navigateur de la victime fait aveuglément confiance au serveur vulnérable.
* **Le mécanisme :** Le serveur renvoie un script malveillant (injecté par l'attaquant). Le navigateur l'exécute car il croit qu'il fait partie légitime de la page web. Le script peut alors lire les cookies et les exfiltrer.

=== Cross-Site Request Forgery (CSRF)
* **La faille :** Le serveur vulnérable fait aveuglément confiance au navigateur de la victime.
* **Le mécanisme :** L'attaquant attire la victime (déjà connectée au site cible) sur une page web malveillante qu'il contrôle. Cette page contient un code (comme un formulaire caché) qui force le navigateur de la victime à envoyer une requête HTTP vers le site cible.
* **Le danger :** Le navigateur joint automatiquement le cookie de session de la victime à cette requête. Le serveur cible reçoit la requête, voit un cookie valide, et exécute l'action (changement de mot de passe, virement bancaire, etc.) en pensant que la victime en est à l'origine.

=== Les Jetons Anti-CSRF (Synchronizer Token Pattern)
C'est la défense historique et la plus robuste.

1. **Génération :** À l'ouverture de la session, le serveur génère une chaîne de caractères aléatoire, longue et imprévisible (le jeton). Il la stocke côté serveur (dans la variable de session `$_SESSION` en PHP).
2. **Transmission :** Chaque formulaire sensible de l'application intègre ce jeton dans un champ caché.
3. **Validation :** Lors de la soumission du formulaire (POST), le serveur compare le jeton reçu avec celui stocké en session. S'ils diffèrent ou si le jeton est absent, la requête est rejetée.


=== Synchronizer Token Pattern : Le token CSRF est stocké en session côté serveur. Le formulaire contient un champ caché avec ce token. À la soumission, le serveur compare les deux.

=== Double Submit Cookie Pattern : Le token est généré côté serveur puis envoyé dans un cookie ET dans un champ caché du formulaire. À la réception, le serveur compare ces deux éléments fournis par le client.
